## 任务背景
在定义数据结构、API 参数、配置文件或其他内容规范时，经常需要清晰表达：
- 哪些字段是可选的
- 哪些字段可以多选或重复
- 哪些字段之间存在条件依赖

不同团队可能使用不同的表示方法（正则表达式、ABNF、JSON Schema等），导致表达混乱。本技能提供一套**简洁、通用的符号系统**，易学易用，跨团队可理解。

## 适用场景
- API 或 SDK 接口文档设计
- 数据库模型或配置文件规范定义
- 表单字段约束描述
- 命令行参数或选项规范
- 内容结构标准化定义
- 知识卡片或笔记模板设计

## 核心符号体系

### 1. 可选（Optional）
表示该字段或内容**可以出现 0 次或 1 次**（有或没有）。

| 符号 | 用途 | 示例 | 说明 |
|------|------|------|------|
| `?` | 后缀可选符 | `颜色?` | 颜色字段可选 |
| `[ ]` | 方括号包裹 | `[年龄]` | 年龄字段可选 |

**推荐**：优先使用 `[ ]` 以保持一致性，`?` 作为补充说明。

---

### 2. 多选与重复（Multiple / Repeatable）

#### 2.1 零或多次（可选且可重复）
表示字段可以**完全不出现、出现一次，或多次**。

| 符号 | 用途 | 示例 | 说明 |
|------|------|------|------|
| `*` | 星号后缀 | `标签*` | 0 个、1 个或多个标签 |
| `...` | 省略号 | `[附件...]` | 附件列表，可选且可重复 |

#### 2.2 一或多次（至少出现一次）
表示字段**至少要出现一次，可多次**。

| 符号 | 用途 | 示例 | 说明 |
|------|------|------|------|
| `+` | 加号后缀 | `成员+` | 至少 1 个成员，可多个 |

#### 2.3 固定选择（单选或多选枚举）
表示从**有限的选项集中选择**。

| 符号 | 用途 | 示例 | 说明 |
|------|------|------|------|
| `{ \|  }` | 花括号 + 竖线 | `{红\|绿\|蓝}` | 从三个颜色中选一个 |
| `( \| )*` | 圆括号 + 竖线 + 星号 | `(游泳\|跑步\|阅读)*` | 可选且可多选 |
| `( \| )+` | 圆括号 + 竖线 + 加号 | `(学生\|在职\|退休)+` | 至少选一个，可多选 |

---

### 3. 条件依赖（Conditional）
表示"当某个条件满足时，该字段才是可选/必填/可见的"。

#### 3.1 三元形式（if-then-else）
```
(条件 ? 内容1 : 内容2)
```

| 示例 | 说明 |
|------|------|
| `(已婚 ? 配偶姓名 : 无)` | 已婚时填配偶姓名，否则无此字段 |
| `(国家="US" ? 邮编长度=5 : 邮编长度=可变)` | 美国格式不同 |

#### 3.2 `if` 条件形式
```
[字段] if 条件
```

| 示例 | 说明 |
|------|------|
| `[出生日期] if 年龄<18` | 当年龄小于18时，需填出生日期 |
| `[发票地址] if 订单类型="B2B"` | 仅 B2B 订单需要发票地址 |

#### 3.3 `when` 形式（更自然的语言）
```
[字段] when 条件
```

| 示例 | 说明 |
|------|------|
| `[紧急联系人] when 有家属` | 有家属时需填 |
| `[续订价格] when 订阅类型="年付"` | 年付类型需要 |

## 完整示例

### 示例 1：个人信息表单
```
基本信息：
  姓名 = <必填字符串>                    // 尖括号表示必填
  年龄? = <正整数>                       // 可选
  
联系方式：
  电话 = {手机|座机}+                    // 至少一个，可多个
  
详细信息：
  [配偶姓名] = (已婚 ? 字符串 : 无)      // 条件依赖
  [子女数] = (已婚 ? 正整数 : 不填)      // 条件依赖
  爱好* = (游泳|跑步|阅读|编程)*         // 0 或多个
```

### 示例 2：API 参数规范
```
GET /products?filters
  category = {衣服|鞋|配件}              // 单选
  [brand]* = 品牌名                      // 可选，可多个
  [minPrice] = 浮点数                    // 价格范围（可选）
  [maxPrice] = 浮点数
  sortBy? = {价格|热度|新品}             // 可选排序
  
POST /orders
  items+ = 商品ID                        // 至少 1 个
  [couponCode] = 代码                    // 可选
  shippingAddress = 地址                 // 必填
  [billingAddress] = (shippingAddress != billingAddress ? 地址 : 无)  // 条件
```

### 示例 3：JSON Schema 映射对照
```
符号格式                    对应 JSON Schema
─────────────────────────────────────────────────
必填字段                    "required": ["field"]
[可选字段]                  不在 required 中
字段*                       "type": "array", "minItems": 0
字段+                       "type": "array", "minItems": 1
{A|B|C}                     "enum": ["A", "B", "C"]
(A|B)*                      "type": "array", "items": {"enum": [...]}
[字段] if 条件              "if": {...}, "then": {...}
```

## 执行流程

### 第一步：理解需求
- 明确定义对象的结构（表单、API、数据库等）
- 列出所有字段和约束条件
- 确定是否涉及条件逻辑

### 第二步：分类字段属性
对每个字段，确定：
- **是否必填**：必填、可选 `[ ]`
- **是否可重复**：单个、多个 `*`、至少一个 `+`、特定选项 `{}`
- **是否有条件**：是否依赖其他字段或值

### 第三步：编写规范
- 优先使用表格形式（清晰、便于文档化）
- 复杂逻辑用伪代码或缩进树形结构
- 同步提供 JSON Schema 或对应代码实现

### 第四步：验证与迭代
- [ ] 每个字段的约束清晰明确
- [ ] 条件逻辑无歧义
- [ ] 与实际代码/文档保持同步
- [ ] 团队成员能理解该规范

## 符号快速参考表

| 需求 | 符号 | 例子 | 含义 |
|------|------|------|------|
| 可选 | `?` 或 `[ ]` | `[年龄]` | 可有可无 |
| 多个（含0个） | `*` | `标签*` | 0、1 或多个 |
| 多个（至少1个） | `+` | `成员+` | 至少 1 个，可多个 |
| 单选 | `{}` | `{红\|绿\|蓝}` | 三选一 |
| 多选（可选） | `()*` | `(爱好)*` | 0 个、1 个或多个爱好 |
| 多选（至少1个） | `()+` | `(技能)+` | 至少 1 个技能 |
| 条件真 | `? :` | `(已婚 ? 配偶名)` | if 已婚 then 需要配偶名 |
| 条件依赖 | `if` | `[字段] if 条件` | 条件满足时该字段出现 |

## 常见陷阱与最佳实践

### ❌ 不推荐做法
- 混合使用多种符号系统（如同时用 `?` 和 JSON Schema 的 `optional`）
- 条件逻辑过于复杂，难以理解
- 只有符号，缺少自然语言说明

### ✅ 推荐做法
- **一致性**：整个文档/项目使用统一的符号集
- **清晰性**：复杂条件用自然语言 + 符号并行表述
- **可追溯**：保留注释说明为什么这样设计
- **工具化**：重要规范考虑生成对应的 JSON Schema、代码验证器等

当使用本技能定义一个新规范时，输出应包含：

1. **名称与背景**：该规范定义的是什么

3. **条件说明**（若有）：逐条解释复杂条件
4. **示例数据**：展示合法和非法的数据样本
5. **映射文档**（可选）：提供对应的 JSON Schema、SQL DDL 等

| 标准 | 何时使用 | 优点 | 缺点 |
|------|---------|------|------|
| **本符号系统** | 快速文档、团队沟通 | 简洁易读、跨语言 | 不够严谨 |

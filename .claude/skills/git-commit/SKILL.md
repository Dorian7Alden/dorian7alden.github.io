# Git 提交 Skill（文档仓库专用）

按照文档仓库提交规范，分析工作区变更并分批提交。

## 提交流程

### 第一步：分析当前状态

并行执行以下命令了解工作区全貌：

```bash
git status
git diff --stat
git diff --cached --stat
git log --oneline -10
```

### 第二步：分组变更文件

根据文件所属的**一级目录**将变更分组。同一目录的变更归入一组，不同目录分开提交。

| 分组依据 | scope |
|----------|-------|
| `old/` 下的文件按原一级目录分组 | 使用原始一级目录名，如 `学习`、`卡片`、`English`、`招聘记录` |
| 根目录 `.md` 文件 | 省略 scope 或按内容主题判断 |
| `.claude/`、`.gitignore` 等配置 | 省略 scope |

新文件（untracked）与对应目录的已修改文件一起提交。

### 第三步：确定 type

根据变更的**操作类型**选择 type：

| type | 判断依据 |
|------|---------|
| `docs` | 新增文档、修改文档内容 |
| `file` | 文件移动、文件重命名 |
| `ref` | 目录结构调整（批量移动、批量删除、目录重组） |

单一文件移动用 `file`，整目录移动/删除用 `ref`。

### 第四步：编写 subject

- 使用中文简述，**只描述操作过程，不描述操作影响或目的**
- 控制在 20 字以内
- **正确示例**（描述操作）：
  - `移动操作系统.md到学习目录`
  - `将所有文档移入old目录`
  - `新增070-073听写练习`
- **错误示例**（描述影响/目的）：
  - ~~`去重23个重复md文件`~~（这是影响，应改为 `删除计算机目录`）
  - ~~`清空根目录以便重新整理`~~（"以便重新整理"是目的，应改为 `将所有文档移入old目录`）

不确定时问自己：这句话是在说"做了什么"还是"为什么做/结果是什么"——选前者。

### 第五步：展示并确认

向用户展示提交计划，格式如下：

```
计划创建 N 个提交：

1. type(scope): <subject>
   - file1.md
   - file2.md
   ...

2. type(scope): <subject>
   - file3.md
   ...
```

等待用户确认后再执行。

### 第六步：逐批执行

用户确认后，按顺序逐批提交：

```bash
git add <file1> <file2> ...
git commit -m "<type>(<scope>): <subject>"
```

每批提交后报告结果。

### 第七步：验证

全部提完后确认：

```bash
git status
git log --oneline -5
```

## 强制规则

1. **不同目录的变更不得混在同一个提交中** — 每个 commit 只包含同一一级目录的文件变更。
2. **commit 格式固定为 `type(scope): 中文subject`** — type 和 scope 必须使用约定值。
3. **subject 只描述操作，不描述影响** — 不写"去重""优化""提升"等效果词，只写具体做了什么操作。
4. **不要跳过钩子** — 不使用 `--no-verify`、`--no-gpg-sign` 等参数。
5. **不要提交敏感文件** — `.env`、`credentials.*`、含密钥/密码的文件需警告用户。

## 边界情况

- **工作区无变更**：告知用户 "工作区干净，无需提交"。
- **只有已暂存（staged）变更**：同样分析分组，按规范提交。
- **只有一种类型的变更**：仍然按规范格式单次提交，无需用户确认直接执行。
- **变更量大无法一笔概括**：说明需要进一步拆分，建议用户先部分暂存。
- **old/ 目录下的文件**：分组时按文件在 old/ 下的原始一级目录归类，scope 使用该原始目录名。

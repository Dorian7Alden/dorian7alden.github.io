## 目录标签化管理规范



> 【强制】：必须遵守的内容
>
> 【推荐】：最好遵守的内容，特殊条件下可以自行调整



### 核心原则

1. 【强制】**目录层级即标签树**：目录路径本身就是分类标签。每一层目录都是一个标签，通过嵌套实现标签的逐步细化。文档的完整标签路径就是它所在的目录路径。

   说明：`学习/Java学习/Spring.md` 表示这篇文档同时带有"学习"和"Java学习"两个标签——"学习"是大的领域标签，"Java学习"是在此领域下的细化标签。目录树即分类体系。

2. 【强制】**根目录干净原则**：仓库根目录下只保留以下文件，其余文档必须放入对应的标签目录中：

   - `README.md` — 仓库说明
   - `CLAUDE.md` — Claude Code 配置
   - `.gitignore` — Git 忽略规则
   - `.docs/` — 文档规范（本目录）
   - `.claude/` — Claude Code 技能配置



### 标签层级设计

3. 【强制】**顶层目录代表独立的大领域**：每个顶层目录应该是一个相对独立、边界清晰的大方向或领域，彼此之间不重叠。

   正例：`学习/`（知识学习）、`项目/`（项目开发）、`Competitions/`（竞赛）——三者互为独立领域。
   反例：同时存在 `Java学习/` 和 `学习/Java学习/`——前者不应该独立为顶层标签。

4. 【推荐】**通过嵌套细化标签**：从顶层领域开始，逐层向下划分子标签。每一层都是对上一层的进一步细分。

   正例：
   ```
   学习/                  ← 标签：学习
   学习/Java学习/          ← 标签：学习 → Java学习
   学习/Java学习/Spring.md ← 标签：学习 → Java学习 → Spring
   ```

5. 【强制】**目录层级上限为 5 层**：超过 5 层的目录结构说明分类体系存在严重问题——要么中间层可以提升为独立顶层标签，要么分类过于零碎需要扁平化。

   【推荐】**日常保持在 3 层左右**：3 层是最佳平衡点——既有足够的分类精度，又不会让路径难以理解。

   层级计数方式：从仓库根目录往下数，每一级目录算 1 层。

   正例（3 层，推荐）：
   ```
   学习/Java学习/Spring.md            ← 2 层目录 + 文件 = 共 3 级
   ```

   正例（4 层，可接受）：
   ```
   非计算机/Daily English Dictation/001-010/Dictation-001.md   ← 3 层目录 + 文件 = 共 4 级
   ```

   反例（6 层，禁止）：
   ```
   学习/Java学习/框架/Spring/核心容器/Bean生命周期.md           ← 5 层目录，共 6 级，过度拆分
   ```
   修正：将中间层扁平化 → `学习/Java学习/Spring-核心容器-Bean生命周期.md`（2 层目录）

6. 【推荐】**不轻易新增顶层目录**：新增顶层目录意味着新增一个独立领域。只有当一批文档确实无法归入现有任何顶层领域，且数量足够支撑一个独立标签时，才创建新顶层目录。



### 标签树设计原则

7. 【推荐】设计标签层级时遵循以下原则：

   - **从宽到窄**：顶层为宽泛领域，底层为具体主题
   - **互斥性**：同一层级的标签之间不重叠
   - **完备性**：任何文档都能在标签树中找到合适位置
   - **稳定性**：标签结构不频繁变动

8. 【推荐】当一个子目录下文档数量过多（如超过 20 篇），或内部主题差异较大时，考虑在子目录内部进一步划分子标签。

   正例：`学习/小林coding/` 下有 12 个面试题文件，但主题包含 Java、MySQL、Redis、操作系统等不同方向，可以按方向进一步划分子目录。

9. 【推荐】**单目录文档数量限制**：单个目录下的直接文件数不宜过多。当已知某类文档会持续积累、数量可能超过一定基数时，应提前用序号范围子目录进行分组，将文件分散。

   - 触发条件：预估文档总量 ≥ 30 篇，或当前目录已有 20+ 个直接文件。
   - 分组方式：创建序号范围子目录，如 `001-010/`、`011-020/`、`021-030/`，每组 10 篇。
   - 目的：避免单目录文件过多导致浏览困难；序号范围本身就是一种轻量标签，暗示文档的先后顺序。

   正例：
   ```
   非计算机/Daily English Dictation/
   ├── 001-010/    ← 第 1-10 期
   ├── 011-020/    ← 第 11-20 期
   ├── 021-030/    ← 第 21-30 期
   └── ...
   ```
   反例：
   ```
   非计算机/Daily English Dictation/
   ├── Dictation-001.md
   ├── Dictation-002.md
   ├── ...（70+ 个文件堆在一个目录下）
   └── Dictation-073.md
   ```



### 文档新增流程

10. 【强制】新增文档时必须沿标签树逐层定位，放入最精确的叶子目录。

   步骤：
   
   1. 确定文档主题
   2. 从顶层标签开始，沿着标签树向下，逐层匹配合适的子标签
   3. 如果现有的子标签都不匹配，文档放在最接近的父标签目录下
   4. 如果有 3 篇以上文档同属一个未定义的子标签，创建该子标签目录
   5. 按 `02-文档命名规范.md` 命名文件



### 文档关联性分析

11. 【强制】分析文档之间的关联性时，按以下优先级依次判断，前一级能确定则不必进入下一级：

    1. **文档名优先**：文件名本身就包含关联信息。如果多个文件名遵循相同的编号体系（如 `01-哈希.md`、`02-双指针.md`、`03-滑动窗口.md`），或共享前缀/后缀模式（如 `Java 基础面试题.md`、`MySQL面试题.md`），则它们的关联已经明确，无需进一步分析内容即可确认归属同一组。

    2. **路径分析次之**：目录路径即标签分类。同一目录下的文档已经通过路径表达了归属关系；父目录与子目录的层级关系表达了标签的从宽到窄的细化关系。路径分析可以确定文档属于哪个体系、与哪些文档共处一个上下文。

    3. **内容分析最后**：仅当前两级信息不足以判断关联时才阅读文档内容。内容分析用于理解文档的实际主题、识别与同级文档的关系、发现命名未能体现的隐藏关联。

    例：
    - 看到 `01-哈希.md` ~ `17-技巧.md` 在同一目录下 → 文档名已表明是编号序列，**不需要逐个打开确认**。
    - 看到 `Java多线程学习手册.md` 在 `计划与反思/` 下，而同目录有 `第一次实习能力要求.md` → 路径表明它属于实习规划体系，**不需要读内容判断归属**。
    - 看到 `蓝桥.md` 无法从文件名确定是哪种内容 → **才需要打开文件**，发现是竞赛经验，归入 `算法题/`。



### 文档迁移流程

12. 【强制】**操作分离原则**：移动文件、重命名文件、修改文档内容三类操作必须分开进行，不得混在同一步骤中。每次只执行一种操作类型，上一步确认无误后再进行下一步。避免同时变动位置和名称导致追溯困难。

13. 【推荐】从 `.old/` 目录迁移文档到正式目录时，逐目录进行，每完成一个顶层目录提交一次。

    步骤：
    
    1. 明确当前要整理的顶层标签
    2. 从 `.old/` 中找到该标签的所有相关文件
    3. 逐一确认文件在标签树中的精确位置
    4. 将确认的文件移到对应的目录路径
    5. 提交 commit（仅移动操作）
    6. 用户确认移动结果后，再进行重命名操作
    7. 提交 commit（仅重命名操作）



### 标签冲突处理

14. 【强制】**高内聚、低耦合原则**：文档分类时尊重文件与其所在目录的内聚关系。有些文件是专门放在对应标签下的，与其所属目录中的其他文件共同构成一个整体（如一个课程体系、一个项目文档集），不要因为该文件涉及某个主题就将其单独抽出。优先维护目录内部的完整性，而非追求单一主题的集中。

    例：`学习/提示词/整理每日笔记.md` 虽然是一篇提示词，但它是"学习"体系的一部分，不应因主题是"提示词"就移入 `AI/提示词/`。

15. 【推荐】当一个文档可能属于多个标签路径时，按以下优先级选择：

    1. **结构化体系优先**：如果文档属于一个有明确结构的学习路径或课程体系（如 Java 框架序列、竞赛训练计划、课程作业集），优先归入该体系，保持体系的完整性。
    2. **内聚性优先**：如果文档与其所在目录的其他文件高度相关、共同构成一个整体，不移出。
    3. 文档的**主要目的**是什么（学习知识？记录问题？提交作业？）
    4. 文档的**受众**是谁（自己复习？提交给老师？面试准备？）
    5. 文档的**时效性**（一次性作业 vs 长期维护的笔记）

    正例：SpringAI 既涉及 AI 又涉及 Java 框架学习，但它是 Java/Spring 生态的一部分，与 Spring、SpringSecurity、MyBatis 等框架共同构成学习体系 → 归入 `学习/Java学习/`。
    反例：把 SpringAI 归入 `AI/学习/`，导致 Java 框架学习体系被拆散。



### 标签合并

16. 【推荐】当两个顶层目录的领域高度重叠时，应合并。合并判断标准：

    - 两个目录的文件互换位置后，目录名仍能准确描述内容
    - 两个目录的归入条件重合度超过 50%
    - 至少一个目录文件数少于 5 篇

    当前建议合并的目录对：
    - `感想/` + `成长记录/` → `反思/`（均为个人反思类）
    - `实习准备/` + `实习计划/` → `实习/`（边界模糊）
    - `项目/` + `我的项目/` → `项目/`（"我的"不增加区分度）

    > 注：`AI资料/` 和 `AI踩坑/` 已合并入新建的 `AI/` 顶层目录。



### 目录说明文件

17. 【强制】每个顶层目录应包含一个 `README.md` 文件，作为该目录的**结构指导说明**。

    内容应包括：
    - 该目录作为标签的含义
    - 子目录结构概览及每级子标签的含义
    - **每个文件的关键词列表**：用 3-5 个关键词标注每个文档的核心内容。关键词必须描述**文档内容本身**（讲了什么知识、讨论了什么问题、记录了哪种经验），而非文档的元数据或外部标签（如平台名、公司名、赛事名、工具名）。格式：`文件名 — 关键词1、关键词2、关键词3`

      正例：`01-哈希.md` → `哈希表、Two Sum、字母异位词`（描述了文档中的算法内容）
      反例：`01-哈希.md` → `LeetCode、Hot100、算法题`（只说了平台和来源，没描述内容）

      正例：`2026-06-09-面试.md` → `面试流程、技术提问、复盘反思`（描述了面试文档中讨论了什么）
      反例：`2026-06-09-面试.md` → `重庆本手、招聘会`（只说公司和场合，没描述内容）
    - **增长预判**：推理每个子目录的内容可能如何增长，预估文档数量上限
    - **结构策略**：基于增长预判，选择合适的目录结构（扁平、编号序列、日期分组、领域分组），必要时预留编号范围或空占位
    - **命名策略**：解释该子目录为什么采用当前的命名格式，以及什么条件下会切换策略

    **维护要求**：修改文档内容后，必须同步更新 README 中该文件的关键词列表，保持文档内容与关键词的一致性。

    目的：让阅读者（或 AI）不仅了解当前结构，更理解设计意图和未来扩展路径。关键词机制减少了每次了解文档都需要打开文件分析内容的需求，大幅提高文档的可读性和维护效率。

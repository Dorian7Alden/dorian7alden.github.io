## 文档撰写规范



> 【强制】：必须遵守的内容
>
> 【推荐】：最好遵守的内容，特殊条件下可以自行调整
>
> 【参考】：经验分享，辅助内容，相关工具等技巧



### 1. 基本格式

1. 【强制】所有文档均使用 Markdown 格式撰写。

   说明：统一文档格式标准，轻量高效，便于 AI 读取，便于管理。

2. 【强制】每个文档中应该给出文档名字（标题）。

   说明：每个文档的第一行内容应该是一个标题，说明本文档的主题，而不是直接开始正文。

3. 【推荐】文档标题（文档名）设置为 2 级标题（`##`）。

   说明：一般情况下使用 `##`。仅仓库根目录下的 `README.md` 文件的文档标题可以设置为 1 级（`#`）。

4. 【强制】内容标题从 3 级标题（`###`）开始。

   说明：文档标题占用 `##`，留给内容的空间从 `###` 开始。标题等级从 3 级开始，字体大小适中，也便于后续迁移文档时灵活升降级。

   正例：

   ```markdown
   
   ## doc-name
   
   
   
   ### Title 1
   
   
   
   #### Subtitle
   
   
   
   content
   
   
   
   #### Subtitle
   
   
   
   content
   
   
   
   ### Title 2
   
   
   
   content
   
   ```

5. 【强制】标题元素的上下文用 1 个空行隔开。

   说明：防止标题大量堆叠，保持排版清晰。

   正例：

   ```markdown
   
   ### Title 1
   
   
   content
   
   
   ### Title 2
   
   
   ### Title 3
   
   ```

6. 【强制】禁用行末添加两个空格 + 回车表示换行。

   说明：统一规范，优雅编辑源码。渲染模式下的换行通过段落间的空行实现。

   反例：

   ```markdown
   line1  
   line2
   ```

   正例：

   ```markdown
   line1
   
   line2
   ```



### 2. 内容分隔

7. 【强制】不同的内容板块之间使用 3 个空行（换行符）分隔。

   说明：Markdown 中，3 个空行在渲染时显示为 1 个空行间隔，将不同板块清晰分隔。

   正例：

   ```markdown
   
   ### 标题 1
   
   line1: content content content content content
   
   line2: content content content content content
   
   
   
   ### 标题 2
   
   line1: content content content content content
   
   line2: content content content content content
   
   ```

8. 【推荐】关联性不强的两个内容块之间用 `---` 分割线分隔。

   说明：分割线与上下文元素之间在渲染模式下应有 1 个空行。

   正例：

   ```markdown
   
   ### Title 1
   
   content 1
   
   
   
   ---
   
   
   
   content 2
   
   
   
   ### Title 2
   
   ```

   反例：（内容很少时不需要分割线，只需换行）

   ```markdown
   content 1 , only a few words
   
   
   
   ---
   
   
   
   content 2 , only a few words
   ```

9. 【推荐】合理使用空行和分割线控制排版，类似写作文时分段落。过度分割适得其反。



### 3. 注释与说明

10. 【推荐】当内容有歧义或不易理解时，使用 `>` 进行注释说明。

    正例：

    ```markdown
    
    ### title 1
    
    > 这个名词的意思是指...
    
    这是一段需要解释的文本。
    
    
    
    ### title 2
    
    这是一段文本。
    
    > 可以理解为...。详细内容参考...
    
    ```



### 4. 中英文混排

11. 【推荐】字母、数字、特殊符号与汉字连接时用一个空格隔开。中文标点不加空格。

    说明：视觉上更加舒适工整。中文符号视觉上自带间距。

    正例：

    ```
    英语字母 alpha 数字 123 标点。beta. 标点符号 . 内联公式 $a + b = c$ 文本
    ```



### 5. 标点符号

12. 【推荐】中文文档使用中文标点符号，英文文档、源代码注释、数学公式使用英文标点。

    正例（写文档时）：

    ```markdown
    正在写文档，这是一段文本内容。我真高兴！有什么问题吗？他说："内容"。
    ```

    正例（写代码注释时）：

    ```python
    # 这是一段注释, 英文标点用逗号
    print("hello world")
    ```

    正例（写数学公式时）：

    ```tex
    \text{若 } x>0,\; y<0,\; \text{则 } f(x,y)=x^2+y^2.
    ```



### 6. 图片

13. 【强制】文档内插入图片时，只能使用在线链接，禁止使用相对路径或绝对路径。

    说明：避免文档迁移、跨平台分享时图片加载失效。

    正例：`![image](https://gitee.com/kualk/pic-go/raw/master/imgs/xxx.png)`
    反例：`![image](images/xxx.png)`



14. 【强制】文档中的图片资源大小不能太大，过大的图片需要压缩后再上传。



15. 【参考】给 Typora 或其他 Markdown 编辑器通过 PicGo 配置 Gitee 个人仓库图床。

    参考教程：https://juejin.cn/post/7089308017266524197



### 7. Dorian-Blog 特有规则

16. 【强制】所有文档必须在文件开头使用 YAML frontmatter 标注创建时间和更新时间：

    ```yaml
    ---
    create-time: 2026-06-13
    update-time: 2026-06-13
    ---
    ```

    说明：
    - `create-time`：文档首次创建的日期
    - `update-time`：文档最后一次修改的日期（首次创建时与 `create-time` 相同）
    - 日期格式统一为 `YYYY-MM-DD`
    - 仅修改格式、修正错别字等微调不需要更新 `update-time`；内容有实质增删改时必须更新

    正例：见上方 YAML 示例。
    反例：无 frontmatter 的文档、只有 `create-time` 没有 `update-time`。

17. 【推荐】卡片的"问题-方案"格式：

    短卡片保持精简（2-6 行），用"问题："和"解决方案："开头，不需要标题层级。

    正例：

    ```markdown
    问题：
    API 调用解析失败，返回 400 错误。

    解决方案：
    检查请求头中的 Content-Type，确保与后端接收格式一致。
    ```

18. 【推荐】代码块明确标注语言类型。

    正例：` ```java `、` ```shell `、` ```python `、` ```text `
    反例：不标注语言的裸 ` ``` `

19. 【推荐】段落应有足够的内容支撑。避免"一个标题 + 一行话"的稀疏结构，除非该段落属于声明性质且有必要存在。

20. 【推荐】学习笔记类文档保持对话式、口语化的中文风格（如"先想个场景"、"简单说"），有助于理解和记忆。竞赛笔记和参考手册类文档可以更正式。
